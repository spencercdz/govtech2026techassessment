# Prompt Analysis

This prompt is designed around operational reliability rather than only conversational quality. The structure begins with identity, mission, and explicit input assumptions so downstream behavior stays consistent when integrated into a production stack. I used scenario-based routing (`buyer_support`, `seller_support`, `new_user_support`, `safety_concerns`) because marketplace support requests are highly role-dependent, and this segmentation improves both response relevance and testability. The prompt also requires pre-response quality checks to reduce policy drift and incomplete guidance.

Safety and escalation are intentionally concrete. Instead of generic "escalate if needed" language, the prompt defines `P0/P1/P2` thresholds with expected actions, including emergency-first behavior for high-risk threats and faster moderation windows for active scams. This makes policy compliance easier to audit in golden tests and helps prevent under-reaction to safety incidents. I also included refusal categories for prohibited assistance and mandated safe alternatives, which preserves usability while enforcing platform boundaries.

Finally, the structured output format enables deterministic evaluation and automation. By requiring fields such as `intent`, `risk_level`, `action_items`, and `escalation`, we can score responses beyond subjective quality and build regression checks for policy-sensitive behavior. University-specific context (academic calendar spikes, dorm constraints, monitored pickup locations) ensures the assistant is not just a generic marketplace bot but one tailored to campus operations.
